Part 1 - What We're Building

Hook is an all-in-one college admin co-pilot. It helps a student handle the non-classroom work of college: planning classes, finding money, spotting opportunities, joining the right communities, and getting directed to campus resources without bouncing across a dozen disconnected websites.

The product story is simple: a student gives Hook one profile, and Hook turns that into the next steps that matter most right now.

AreaWhat the student seesWhy it matters
AcademicDegree progress, prerequisite warnings, next-semester recommendations, eligible classes, professor-signal dataStudents stop guessing whether they are on track and what to take next.
ScholarshipsFit-scored scholarships ranked by relevance and urgencyStudents stop manually digging through portals and missing deadlines.
ResearchFit-scored research opportunities with filters and full detailsStudents can discover labs and programs that actually match their background.
InternshipsFit-scored internship and new grad roles with filters and detail viewsStudents see career opportunities without treating LinkedIn as a second full-time job.
ClubsRanked student organizations based on interests, goals, and time commitmentStudents find communities that fit them instead of scrolling through long org lists.
ChatProfile-aware assistant for outreach drafts, study strategy, campus resources, and general UT questionsStudents have one place to ask for help when life gets messy.
The pitch in one line Hook reduces college administrative stress by turning one student profile into clear academic, financial, opportunity, campus, and community next steps.
How this MVP should feel The goal is not to prove that every dataset is live and perfect. The goal is to prove the product shape: one profile in, a coordinated support layer out. Use curated data and a polished flow to sell the vision.

Part 2 - Scope Lock

This section keeps the build honest. Three people makes full scope realistic, but only if you stay disciplined about what is real, what is curated, and what is deferred.

BucketWhat belongs here
Real in MVPOnboarding, dashboard, page navigation, deterministic matchers, fit scores, filters, detail views, chat UI, profile-aware chat backend, docs, screenshots, deployment
Curated / mock-backedDegree plans, course catalog metadata, professor-signal data, scholarships, internships, research roles, clubs, campus resources, sports snapshot
DeferredLive scraping, auth, university integrations, live application submission, production trust and safety, live sports data
Do not accidentally build the production version If a task starts drifting into scraping, auth, or real integration work, stop. The hackathon goal is a convincing, working MVP. Curated local data is not cheating here. It is the correct scope decision.

Part 3 - Codex + Worktrees Explained Simply

3.1 What is Codex?

Codex is an AI coding agent. You give it a precise build instruction and it creates files, writes code, updates docs, and helps with repetitive engineering work. Think of it like a fast implementation partner that still needs a human reviewing the work.

The winning mindset Codex performs best when your prompts are specific. Do not say "make an academic page." Say exactly what files to create, what data shape to expect, what the UI should show, and what docs need to be updated.

3.2 What is a worktree?

A worktree is a second copy of the same repository on a different branch. With three people, this means all builders can work at the same time without stepping on each other.

PersonBranchDirectoryOwns
Person Amainrepo rootPlatform shell, chat, docs, integration
Person Bfeat/academic-clubs../hook-academic-clubsAcademic and clubs
Person Cfeat/opportunities../hook-opportunitiesScholarships, research, internships

3.3 Why the judges care

The Codex story is worth a quarter of the rubric. Three people using three parallel worktrees, with clear ownership and screenshot evidence, is a clean, believable example of AI-assisted product development.

OpenAI productHow Hook uses itWhy it matters in the pitch
CodexBuilds code, data files, docs, and structured UI in parallel worktreesShows how three students can ship much more than a normal student team
GPT-4oPowers profile-aware chat, outreach drafting, and flexible support answersShows Hook can act on a student's context, not just display static data

3.4 How to get into Codex

1
Open Codex - Use the Codex app or platform.openai.com/codex and connect the repo.
2
Pick the correct branch - Person A runs tasks on main. Person B runs tasks on feat/academic-clubs. Person C runs tasks on feat/opportunities.
3
Paste one prompt at a time - Let Codex finish, review the diff, commit, push, then move to the next prompt.
4
Always save evidence - Take screenshots of the prompt, diff, and three active worktrees. This becomes part of the pitch.

Part 4 - Environment Setup

Do this before serious building starts. Setup work is not glamorous, but every minute you save here becomes demo polish later.

4.1 One-Time Setup

  • Install Node.js 20+, Git, and VS Code.
  • Create the GitHub repository and add all three teammates.
  • Make sure all three people can access Codex.
  • Make sure at least one person has an OpenAI API key for local testing and Vercel deployment.
  • Create an empty Next.js app before the hackathon or immediately at the start.

4.2 Worktree Commands

# Person A - clone the repo and stay on main
git clone <repo-url> hook
cd hook
git checkout main

# Create the other two branches and worktrees
git branch feat/academic-clubs
git branch feat/opportunities
git worktree add ../hook-academic-clubs feat/academic-clubs
git worktree add ../hook-opportunities feat/opportunities

# Person B works in the academic/clubs directory
cd ../hook-academic-clubs
git branch

# Person C works in the opportunities directory
cd ../hook-opportunities
git branch
Plain English After this, you now have three copies of the same repo. One copy is for Person A. One is for Person B. One is for Person C. You can all build at the same time without waiting on each other.

8.2 Plain-English Conflict Resolution

1
Check which file is broken - Run git status. Any file marked as conflicted needs attention.
2
Open the file and look for conflict markers - They look like <<<<<<<, =======, and >>>>>>>.
3
Keep the code that makes sense - Usually the right answer is "keep both changes, but in a clean order."
4
Then re-test - Run npm run build after every merge. Do not assume the merge is safe just because Git accepted it.
How to prevent conflicts Person A owns shared files like lib/types.ts and app/dashboard/page.tsx. Persons B and C should not hot-edit those files directly. If a shared type needs a change, ask Person A to make it.

Part 13 - Emergency Playbook

ProblemSymptomsFix
Codex task fails or stalls The task errors or never finishes Re-run once with the same prompt. If it fails again, shrink the scope of that prompt and move on. Do not lose 20 minutes on one stuck task.
Type errors after merge npm run build fails Check shared types first. Most merge issues come from field name mismatches or import paths.
Dashboard cards are empty Pages work individually, but the dashboard shows no data Switch on DEMO_STATIC or route the dashboard through lib/demo-fallbacks.ts until the live API issue is fixed.
Chat gives vague answers Responses feel generic or ignore the profile Make the system prompt include the StudentProfile JSON and the relevant local resource snippets. Keep the answer contract short and structured.
Vercel deploy fails Build logs error out Check OPENAI_API_KEY, then run npm run build locally and fix the first error only. Do not chase all errors at once.
Merge conflict in a shared file Git reports CONFLICT in main Person A resolves it. If needed, abort the merge, have Person B and/or Person C re-push the latest branches, and re-merge cleanly.
Demo laptop dies You lose the local environment Use the deployed URL or another laptop. Keep the Vercel URL and a backup recording on all three teammates' phones.
